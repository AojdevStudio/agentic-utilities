# Setup Ossie's Agent Skills

Scaffold the per-repo configuration that the engineering skills assume:

- **Issue tracker** — where issues live (GitHub by default; local markdown is also supported out of the box)
- **Agent workflow** — how work moves from plan/spec to issues, project board, PR, review feedback, merge, and release
- **Triage labels** — the strings used for the five canonical triage roles
- **Project context docs** — where `CONTEXT.html` or `CONTEXT.md`, ADRs, `PRODUCT.md`, and `DESIGN.md` live, and the consumer rules for reading them

This is a prompt-driven skill, not a deterministic script. Explore, present what you found, confirm with the user, then write. The goal is not just to add docs; the goal is to make the repo operable by AI agents through the same tools humans already use.

## Operating principles

- Prefer the repo's real remote workflow over local scratch state when it exists. A GitHub repo should usually use GitHub Issues, GitHub Projects, PRs, review comments, and Actions rather than markdown task files.
- For GitHub-backed repos, GitHub Issues and a linked GitHub Project board are the default work hub. Use local markdown only when the user explicitly chooses it or GitHub auth/repo access is blocked.
- Treat CLI access as part of the setup surface. If the repo uses GitHub, record whether `gh` works and whether auth is available; if it is not available, write the docs with a clear auth blocker.
- Encode the whole loop, not just the issue store: plan/spec -> issues -> one issue at a time -> branch/PR -> self-review -> automated review -> address/reply -> merge/cleanup -> update issue/project status -> release when relevant.
- Keep generated docs short and operational. Long explanations belong in referenced docs only when the repo actually needs them.

## Process

### 1. Explore

Look at the current repo to understand its starting state. Read whatever exists; don't assume:

- `git remote -v` and `.git/config` — is this a GitHub repo? Which one?
- If this is a GitHub repo, run non-destructive GitHub CLI checks:
  - `gh auth status` — is CLI auth available?
  - `gh repo view --json nameWithOwner,description,visibility,defaultBranchRef` — what repo are agents operating on?
  - `gh issue list --limit 20 --json number,title,state,labels` — what issue vocabulary already exists?
  - `gh label list --limit 100 --json name,description` — what labels should be reused instead of duplicated?
  - `gh project list --owner <owner> --limit 20` — is there a GitHub Project board to use?
  - If no project exists and auth is valid, create a repo backlog project and link it to the repo. Record the project owner, number, title, and URL in `docs/agents/workflow.md`.
- `AGENTS.md` and `CLAUDE.md` at the repo root — does either exist? Is there already an `## Agent skills` section in either?
- `CONTEXT.html`, `CONTEXT.md`, `CONTEXT-MAP.html`, and `CONTEXT-MAP.md` at the repo root
- `docs/adr/` and any `src/*/docs/adr/` directories
- Impeccable context lookup locations for `PRODUCT.md` and `DESIGN.md`: repo root first, then `.agents/context/`, then `docs/`
- `.impeccable/design.json` if present — optional sidecar design-system data generated by `impeccable document`
- `docs/agents/` — does this skill's prior output already exist?
- `.scratch/` — sign that a local-markdown issue tracker convention is already in use
- `.github/workflows/` — does the repo already have CI, review, deploy, or release workflows?

### 2. Present findings and ask

Summarise what's present and what's missing. Then walk the user through the four decisions **one at a time** — present a section, get the user's answer, then move to the next. Don't dump all four at once.

Assume the user does not know what these terms mean. Each section starts with a short explainer (what it is, why these skills need it, what changes if they pick differently). Then show the choices and the default.

**Section A — Issue tracker.**

> Explainer: The "issue tracker" is where issues live for this repo. Skills like `to-issues`, `triage`, `to-prd`, and `qa` read from and write to it — they need to know whether to call `gh issue create`, write a markdown file under `.scratch/`, or follow some other workflow you describe. Pick the place you actually track work for this repo.

Default posture: these skills were designed for GitHub. If a `git remote` points at GitHub, propose that. If a `git remote` points at GitLab (`gitlab.com` or a self-hosted host), propose GitLab. Otherwise (or if the user prefers), offer:

- **GitHub** — issues live in the repo's GitHub Issues (uses the `gh` CLI)
- **GitLab** — issues live in the repo's GitLab Issues (uses the [`glab`](https://gitlab.com/gitlab-org/cli) CLI)
- **Local markdown** — issues live as files under `.scratch/<feature>/` in this repo (good for solo projects or repos without a remote)
- **Other** (Jira, Linear, etc.) — ask the user to describe the workflow in one paragraph; the skill will record it as freeform prose

**Section B — Agent workflow surface.**

> Explainer: The issue tracker says where work is stored. The "workflow surface" says how an agent should move work through the system: whether to use a GitHub Project board, whether every issue needs a PR, how review comments are fetched and answered, and whether release/deploy pipelines are part of the agent's job. Without this, agents know where tickets live but not how work actually gets finished.

Default posture: if the repo is on GitHub, use **GitHub Issues + Project board + PRs**. If no linked GitHub Project exists and `gh auth status` passes, create/link one as part of setup. If auth or permissions block project creation, record the blocker in `docs/agents/workflow.md` and do not fall back to local markdown unless the user explicitly chooses that.

- **GitHub Issues + Project board + PRs** — same as above, plus agents update project status as work moves.
- **GitHub Issues + PRs only** — allowed only when the user explicitly declines Project boards or GitHub Projects are unavailable.
- **Local markdown workflow** — specs/tasks stay under `.scratch/`; useful only when there is no remote tracker.
- **Other** — ask the user to describe the lifecycle from task creation to review/merge/release.

Ask which review and release conventions to record:

- Should agents self-review before opening/updating a PR?
- Should agents fetch and answer PR review comments with the tracker CLI?
- Should agents wait for, request, or summarize automated reviews?
- Should agents manage release/deploy workflows, or only mention them when asked?

**Section C — Triage label vocabulary.**

> Explainer: When the `triage` skill processes an incoming issue, it moves it through a state machine — needs evaluation, waiting on reporter, ready for an AFK agent to pick up, ready for a human, or won't fix. To do that, it needs to apply labels (or the equivalent in your issue tracker) that match strings *you've actually configured*. If your repo already uses different label names (e.g. `bug:triage` instead of `needs-triage`), map them here so the skill applies the right ones instead of creating duplicates.

The five canonical roles:

- `needs-triage` — maintainer needs to evaluate
- `needs-info` — waiting on reporter
- `ready-for-agent` — fully specified, AFK-ready (an agent can pick it up with no human context)
- `ready-for-human` — needs human implementation
- `wontfix` — will not be actioned

Default: each role's string equals its name. Ask the user if they want to override any. If their issue tracker has no existing labels, the defaults are fine.

**Section D — Project context docs.**

> Explainer: Several skills read project context before working. Architecture and debugging skills read `CONTEXT.md` for domain language and ADRs for decisions. UI/design skills such as `impeccable` read `PRODUCT.md` for strategy and `DESIGN.md` for the visual system. Record where these files live so agents do not design, diagnose, or refactor from generic assumptions.

Confirm the layout:

- **Single-context** — one `CONTEXT.md` + `PRODUCT.md` + `DESIGN.md` + `docs/adr/` at the repo root. Most repos are this.
- **Multi-context** — `CONTEXT-MAP.md` at the root pointing to per-context `CONTEXT.md` files, with product/design context either at the repo root or in `.agents/context/` / `docs/`.

For `PRODUCT.md` / `DESIGN.md`, use the same lookup standard as `impeccable`: root first, then `.agents/context/`, then `docs/`. Record any `IMPECCABLE_CONTEXT_DIR` override if the repo uses one. Treat `PRODUCT.md` as required for design work and `DESIGN.md` as optional but strongly recommended.

### 3. Confirm and edit

Show the user a draft of:

- The `## Agent skills` block to add to whichever of `AGENTS.md` / `CLAUDE.md` is being edited (see step 4 for selection rules)
- The contents of `docs/agents/issue-tracker.md`, `docs/agents/workflow.md`, `docs/agents/triage-labels.md`, `docs/agents/domain.md`

Let them edit before writing.

### 4. Write

**Pick the file to edit:**

- If `AGENTS.md` exists, edit it.
- Else if `CLAUDE.md` exists, edit it.
- Else create `AGENTS.md`.

Never create `CLAUDE.md` when `AGENTS.md` already exists (or vice versa) — edit the existing instruction file, preferring `AGENTS.md` for Codex repos.

If an `## Agent skills` block already exists in the chosen file, update its contents in-place rather than appending a duplicate. Don't overwrite user edits to the surrounding sections.

The block:

```markdown
## Agent skills

### Issue tracker

[one-line summary of where issues are tracked]. See `docs/agents/issue-tracker.md`.

### Agent workflow

[one-line summary of issue/project/PR/review/release workflow]. See `docs/agents/workflow.md`.

### Triage labels

[one-line summary of the label vocabulary]. See `docs/agents/triage-labels.md`.

### Project context docs

[one-line summary of domain/product/design layout — "single-context" or "multi-context"]. See `docs/agents/domain.md`.
```

Then write the docs files using the seed templates in this skill folder as a starting point:

- [issue-tracker-github.md](./issue-tracker-github.md) — GitHub issue tracker
- [issue-tracker-gitlab.md](./issue-tracker-gitlab.md) — GitLab issue tracker
- [issue-tracker-local.md](./issue-tracker-local.md) — local-markdown issue tracker
- [workflow.md](./workflow.md) — agent operating loop, project board, PR review, and release conventions
- [triage-labels.md](./triage-labels.md) — label mapping
- [domain.md](./domain.md) — domain doc consumer rules + layout

For "other" issue trackers, write `docs/agents/issue-tracker.md` from scratch using the user's description.

### 5. Done

Tell the user the setup is complete and which engineering skills will now read from these files. Mention they can edit `docs/agents/*.md` directly later — re-running this skill is only necessary if they want to switch issue trackers, change the agent workflow surface, or restart from scratch.
